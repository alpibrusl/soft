**LLM provider config** (#196 / PR #203) is the headline. `agent.local_complete` and `agent.cloud_complete` now make real HTTP calls. Configured purely via env vars:

- `OLLAMA_HOST` (default `http://localhost:11434`), `LEX_LLM_LOCAL_MODEL` (default `llama3`).
- `LEX_LLM_CLOUD_BASE_URL` (defaults to OpenAI; supports Mistral, Groq, Together, DeepSeek, vLLM), `LEX_LLM_CLOUD_MODEL`, `LEX_LLM_CLOUD_API_KEY` (or `OPENAI_API_KEY` fallback).
- Custom auth / batching / fallback via the `EffectHandler` trait (the escape hatch we asked for).

## 2. What soft wired in this update

### 2.1 Workspace pinned to v0.2.2

`cargo update` brought all nine lex-lang crates from 0.2.1 → 0.2.2. Workspace builds clean; 23 test suites all green.

### 2.2 LLM-driven agent ([`agents/llm_driver.lex`](../../agents/llm_driver.lex))

A new agent definition that calls `agent.local_complete` from a Lex handler and forwards the model's response over A2A. Compile-time `[llm_local]` effect on the handler return type promotes the call from "type error" to "allowed." Demonstrates the new wires end-to-end:

```lex
import "std.agent" as agent

fn config() -> AgentConfig {
  agent_new("llm_driver")
  |> agent_peers(["logger"])
  |> agent_effects(["llm_local", "a2a"])
  |> agent_handles([{ topic: "Decide", fn_name: "on_decide" }])
}

fn on_decide(state, _msg) -> [llm_local] List[ActionRecord] {
  let answer := unwrap_or(agent.local_complete(state.prompt), "(llm unavailable)")
  [{ kind: "send_a2a", peer: "logger", a2a_topic: "Decision",
     payload_json: answer, ... }]
}
```

Verified manually: `soft-run agents/llm_driver.lex --port 8005 --state-json '{"prompt":"hi"}'`, then `curl … /a2a/messages` with a `Decide` message. Without Ollama running, the call returns `Err`; `unwrap_or` falls back to a sentinel string. With Ollama running, the model's response flows out as a SendA2a payload.

### 2.3 DSL preamble + `.lex` files now use `_param` for unused params

PR #205's underscore support means our handler signatures can write `_msg :: { ... }` instead of `msg` when ignored. Cleaner intent; no behaviour change. Already adopted in `agents/llm_driver.lex`.

## 4. What we'd consider asking for next (low-priority)

- **Record-update spread syntax** — `{ ..base, peers: x }`. Our DSL preamble's `agent_*` builders each have to copy all 7 fields by hand. Would shrink the preamble from ~80 lines to ~20.
- **`std.json.parse[T]`** — depot's `on_pv_update` currently bumps `pv_kw` by a fixed delta because we can't parse a numeric value out of a JSON-encoded payload string. Working around it is fine for now.
- **Effect-typed parameterization** (`[mcp(<server>)]`) — would let us promote per-server MCP scoping from runtime to compile-time. soft-agent's runtime allowlist works fine today; this would just be a typing improvement.

None blocking.

## 5. Where the project sits

Counting today (v9 of soft-agent + v0 of soft-a2a + v0 of soft-runner):

- Workspace: 23 test suites, all green.
- Five agent types defined in pure Lex: `vehicle`, `depot`, `tms`, `pv`, `llm_driver`. Plus `depot1`/`depot2` variants for cross-depot fall-over.
- Multi-process runner (`crates/soft-runner` → `soft-run` binary): each agent runs as its own OS process, communicating over real A2A HTTP.
- Multi-process deployment scripted via `deploy/Procfile` + `deploy/run-local.sh`.
- All four lex-lang follow-ups (#196, #197, #198, parser papercut) closed.
- Soft side has implementations for: tick mechanism, stateful Lex handlers, in-process router, A2A-routed executor, soft-replay binary, agent DSL preamble, real LLM call from Lex code.

**Phase 1 demo is achievable end-to-end with what's in the repo.** The only piece blocked on external work is hardware verification of the aarch64 binaries.
